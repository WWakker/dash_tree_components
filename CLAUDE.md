## Commands

### Setup
- `npm install` — install JS deps
- `python -m venv venv && . venv/bin/activate && pip install -r requirements.txt` — install Python build deps (needs `dash[dev]>=4` for `dash-generate-components`)
- `pip install -r tests/requirements.txt` — extra deps for Selenium tests (includes `chromedriver-autoinstaller`, so no manual driver install needed)

### Build
- `npm run build` — full build: webpack bundle **and** regenerate Python/R/Julia wrappers. Requires the venv to already be active.
- `npm run build:activated` — same as `build` but activates `./venv` for the backend step (use this if you're not already in the venv).
- `npm run build:js` — webpack only; emits `dash_tree_components/dash_tree_components.min.js` into the Python package dir. Use this when you've changed only JSX or CSS and PropTypes are untouched.
- `npm run build:backends` — `dash-generate-components` only; rewrites `dash_tree_components/Tree.py`, `_imports_.py`, `metadata.json`, `package-info.json`, the R wrappers (`R/*.R`, `man/*.Rd`, `NAMESPACE`, `DESCRIPTION`, `inst/deps/`), and the Julia wrappers (`Project.toml`, `deps/`). Use this after editing PropTypes or JSDoc.

### Run
- `npm start` — webpack-dev-server with the pure-React demo from `src/demo/` (no Dash, no Python). Fast iteration on `Tree.react.js`/`Node.react.js`.
- `python usage.py` — runs the Dash demo at http://localhost:8050. Requires `npm run build` first so the Python wrapper and JS bundle are up to date.

### Test
- `pytest tests --headless` — Selenium integration tests against `usage.py`. The `tests/conftest.py` runs `chromedriver_autoinstaller.install()` on session start, so no PATH setup is required as long as Chrome is installed.
- `pytest tests/test_usage.py::test_search_filters_to_matches --headless` — single test. Drop `--headless` to see the browser.

### Publish
- `npm run validate-init` (auto-run on `npm publish` via `prepublishOnly`) checks every file in `dash_tree_components/` is listed in `MANIFEST.in` and the `_js_dist`/`_css_dist` in `__init__.py`.
- Version is read from `package.json` by `setup.py` — bump it in `package.json` only.

## Architecture

This is a Plotly Dash component library generated from `dash-component-boilerplate`. The single exported component is `Tree`, a recursive, non-virtualized hierarchical tree with CSS-animated expand/collapse, full WAI-ARIA `tree` keyboard support, and Dash-controllable selection and expansion state. The only runtime React dependency is `react-icons`.

### Source-of-truth layout

```
src/lib/components/Tree.react.js    ← top-level Tree (state, keyboard, ARIA wrapper)
src/lib/components/Node.react.js    ← recursive per-row renderer (no PropTypes — internal)
src/lib/index.js                    ← exports Tree
src/lib/styles.css                  ← all component styles, scoped to .tree-container
src/demo/                           ← pure-React playground for `npm start`

dash_tree_components/               ← Python package; mostly AUTOGENERATED
  Tree.py                           ← DO NOT EDIT — regenerated from Tree.react.js PropTypes/JSDoc
  _imports_.py                      ← DO NOT EDIT — regenerated
  __init__.py                       ← HAND-MAINTAINED — wires _js_dist/_css_dist
  dash_tree_components.min.js[.map] ← webpack output

R/, man/, NAMESPACE, DESCRIPTION,
inst/deps/, Project.toml, src/jl/   ← autogenerated R + Julia bindings

tests/conftest.py                   ← runs chromedriver_autoinstaller.install()
tests/test_usage.py                 ← Selenium tests driving usage.py via dash_duo
```

### Component design

`Tree` is **non-virtualized**: it renders every node in `data` recursively. It used to wrap `react-arborist`; that was dropped because `react-arborist`'s absolute-positioned rows can't be animated on mount/unmount. The trade-off is real: this implementation is unsuitable for trees with many thousands of rows. For typical Dash table-of-contents use (≤ a few hundred leaves) it's the better fit and supports proper expand/collapse animations.

Key responsibilities live in `Tree.react.js`:
- **State**: search term, expanded folder set, active row (roving tabindex target).
- **Keyboard handler**: single `onKeyDown` delegated on `.tree-scroll` (catches events from any focused descendant via bubbling).
- **Visible-row computation**: `useMemo` flattens the tree honoring search filter + expanded set into a list keyboard nav can index into.
- **Focus management**: `rowRefs` map of `id → HTMLLIElement`, populated via callback refs on `<Node>`. `focusRow(id)` calls `.focus()` synchronously plus a `requestAnimationFrame` re-focus to defeat the browser's hash-navigation focus-shift on `<a href="#…">` leaf clicks.
- **Dash wiring**: `setProps({selected_id})` on click/Enter, `setProps({expanded_ids})` on toggle. External changes to those props sync back into state via effects with content-equality checks to prevent echo loops.

`Node.react.js` is a pure render component receiving a `tree` API object from its parent. It carries no local state — open/closed is owned by `Tree` so the keyboard navigator can compute the visible flat list.

### Expand/collapse animation

CSS-only via the `grid-template-rows: 0fr ↔ 1fr` interpolation trick. The `.tree-children` wrapper is `display: grid` with `grid-template-rows: 0fr` when closed and `1fr` when open; the inner `.tree-children-inner` has `overflow: hidden; min-height: 0`. Transitioning between `0fr` and `1fr` animates the row height from zero to natural content height without any JS measurement. Browser support: modern Chrome/Firefox/Safari. There's no `prefers-reduced-motion` guard (open issue).

### ARIA / keyboard

Implements the [WAI-ARIA tree pattern](https://www.w3.org/WAI/ARIA/apg/patterns/treeview/):

- Outer `<ul>`: `role="tree"`, `aria-label` (from the `aria_label` prop, default `"Tree"`).
- Each row's `<li>`: `role="treeitem"`, `aria-level`, `aria-posinset`, `aria-setsize`, `aria-selected`, `aria-expanded` (folders only — omitting on leaves is required).
- Each folder's children `<ul>`: `role="group"`.
- **Roving tabindex**: exactly one row has `tabIndex=0`; the rest have `-1`. Leaf `<a>` elements are `tabIndex=-1` so they don't pollute the tab order.

Keyboard shortcuts handled in `Tree.react.js`'s `handleKeyDown`:

| Key | Effect |
|---|---|
| `ArrowDown` / `ArrowUp` | Next / previous visible row |
| `ArrowRight` | Open closed folder; if open, move to first child |
| `ArrowLeft` | Close open folder; if closed/leaf, move to parent |
| `Home` / `End` | First / last visible row |
| `Enter` / `Space` | Select, toggle folders, navigate leaf `href`s |

### The autogeneration flow

`npm run build` does two things:

1. **webpack** (`build:js`) bundles `src/lib/index.js` into `dash_tree_components/dash_tree_components.min.js`. The webpack `output.path` writes directly into the Python package dir.
2. **`dash-generate-components`** (`build:backends`) reads PropTypes and JSDoc comments from the React source and emits the Dash wrappers for Python, R, and Julia. The script is invoked with `--r-prefix '' --jl-prefix ''` so the R/Julia function names are not prefixed.

Implication: to change a prop, its default, or its docstring, edit **`Tree.react.js`** (PropTypes + JSDoc), then re-run `npm run build`. Never edit `Tree.py` directly.

`Node.react.js` intentionally has no PropTypes — it's an internal helper, not a Dash component. `dash-generate-components` prints a `TypeError` for it on each build; that's expected noise. It still completes and generates `Tree.py`.

### Prop naming convention

React props use **snake_case** (`row_height`, `open_by_default`, `padding_top`, `collapse_icon_color`, `expanded_ids`, `selected_id`, `aria_label`) because the generator preserves the JS prop name in the Python API and Dash users expect Pythonic kwargs. The exceptions are `className` and `rowClassName`, which mirror the React DOM convention. Keep new props snake_case unless they're a direct passthrough to a camelCase library prop. DOM-attribute props with hyphens (e.g. `aria-label`) must use a Python-friendly alias like `aria_label` because `dash-generate-components` writes the prop name verbatim into a Python keyword argument.

### Demo vs. Dash runtime

`src/demo/App.js` simulates Dash's runtime by maintaining local state and passing a hand-rolled `setProps` to `<Tree>`. The real `setProps` is injected by Dash at runtime. If you add a new prop that the component writes back (i.e. a controlled prop reported to Dash callbacks), make sure the demo's `setProps` stub still merges it correctly (the current stub does `setState(newProps)` — a full overwrite — which works because the demo doesn't depend on prior state being preserved).

### Bundle-asset wiring

`dash_tree_components/__init__.py` builds the `_js_dist` list that tells Dash which JS files to serve. If you add a new chunk (e.g. via webpack `splitChunks` async groups or a CSS file), it must be:

1. Added to `_js_dist` (or `_css_dist`) in `__init__.py`,
2. Listed in `MANIFEST.in` so it's included in the sdist/wheel.

`_validate_init.py` enforces both at publish time.

### Tree data shape

`data` is a list of nodes. Each node has:
- `id` (string, required, must be unique across the entire tree — used as the React `key` and for `selected_id` / `expanded_ids` referencing).
- `name` (display text).
- `children` (array, optional — presence determines folder vs. leaf).
- `href` (string, optional — rendered as `<a href>` on leaves; clicking navigates and the row stays selected).
- `icon_color` (string, optional — overrides `node_icon_color` for that node).

See `usage.py` and `src/demo/data.js` for examples.

### Controlled props worth knowing about

- `selected_id` (string): updated via `setProps` on click/Enter; settable from Dash. Setting it externally auto-expands every ancestor folder and scrolls the row into view.
- `expanded_ids` (list of strings): updated via `setProps` on every toggle; settable from Dash. Initial value derives from `open_by_default` if not provided — the component pushes the derived initial set to Dash on mount so callbacks see a real value rather than `null`.

The sync-from-prop effects use content-equality checks (`setsEqual`) so the `setProps → new prop → effect` round-trip doesn't re-trigger state updates and loop.

## Known issues and trade-offs

- **No virtualization**: don't use for trees with thousands of rows.
- **`prefers-reduced-motion` not respected**: the grid animation always runs.
- **Selection / focus ring colors are hard-coded** in `styles.css`. No prop yet.
- **Search is naive**: no debounce, no match highlight.
- **Bundle is ~4.5 MiB**, mostly `react-icons` SVG data. Five icons are actually used — inlining them as JSX would shrink the bundle ~80%.
- **`Node.react.js` PropTypes-less** means `dash-generate-components` prints a `TypeError` per build (cosmetic, doesn't block).
