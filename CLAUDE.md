# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

### Setup
- `npm install` — install JS deps
- `python -m venv venv && . venv/bin/activate && pip install -r requirements.txt` — install Python build deps (needs `dash[dev]` for `dash-generate-components`)
- `pip install -r tests/requirements.txt` — extra deps for Selenium tests

### Build
- `npm run build` — full build: webpack bundle **and** regenerate Python/R/Julia wrappers. Requires the venv to already be active.
- `npm run build:activated` — same as `build` but activates `./venv` for the backend step (use this if you're not already in the venv).
- `npm run build:js` — webpack only; emits `dash_tree_components/dash_tree_components.min.js` into the Python package dir.
- `npm run build:backends` — `dash-generate-components` only; rewrites `dash_tree_components/Tree.py`, `_imports_.py`, `metadata.json`, `package-info.json`, the R wrappers (`R/*.R`, `man/*.Rd`, `NAMESPACE`, `DESCRIPTION`, `inst/deps/`), and the Julia wrappers (`Project.toml`, `deps/`).

### Run
- `npm start` — webpack-dev-server with the pure-React demo from `src/demo/` (no Dash, no Python). Useful for fast iteration on `Tree.react.js`.
- `python usage.py` — runs the Dash sample at http://localhost:8050. Requires `npm run build` first so the wrapper and bundle are up to date.

### Test
- `pytest tests` — Selenium integration tests against `usage.py`. **Heads-up:** `tests/test_usage.py` is still the boilerplate stub (selects `#input > input`, which doesn't exist in this component) and will fail until rewritten for the actual `Tree` API.
- `pytest tests/test_usage.py::test_render_component` — single test.

### Publish
- `npm run validate-init` (auto-run on `npm publish` via `prepublishOnly`) checks every file in `dash_tree_components/` is listed in `MANIFEST.in` and the `_js_dist`/`_css_dist` in `__init__.py`.
- Version is read from `package.json` by `setup.py` — bump it in `package.json` only.

## Architecture

This is a Plotly Dash component library generated from `dash-component-boilerplate`. The single component is `Tree`, a hierarchical/searchable tree built on top of [`react-arborist`](https://github.com/brimdata/react-arborist).

### Source-of-truth layout

```
src/lib/components/Tree.react.js    ← React source (edit this)
src/lib/components/Node.react.js    ← per-row renderer (edit this)
src/lib/index.js                    ← exports the component(s)
src/demo/                           ← pure-React playground for `npm start`

dash_tree_components/               ← Python package; mostly AUTOGENERATED
  Tree.py                           ← DO NOT EDIT — regenerated from Tree.react.js PropTypes/JSDoc
  _imports_.py                      ← DO NOT EDIT — regenerated
  __init__.py                       ← HAND-MAINTAINED — wires _js_dist/_css_dist
  dash_tree_components.min.js[.map] ← webpack output

R/, man/, NAMESPACE, DESCRIPTION,
inst/deps/, Project.toml, deps/     ← autogenerated R + Julia bindings
```

### The autogeneration flow

`npm run build` does two things:

1. **webpack** (`build:js`) bundles `src/lib/index.js` into `dash_tree_components/dash_tree_components.min.js`. The webpack `output.path` writes directly into the Python package dir.
2. **`dash-generate-components`** (`build:backends`) reads PropTypes and JSDoc comments from the React source and emits the Dash wrappers for Python, R, and Julia. The script is invoked with `--r-prefix '' --jl-prefix ''` so the R/Julia function names are not prefixed.

Implication: to change a prop, its default, or its docstring, edit **`Tree.react.js`** (PropTypes + JSDoc), then re-run `npm run build`. Never edit `Tree.py` directly.

### Prop naming convention

React props use **snake_case** (`row_height`, `open_by_default`, `padding_top`, `collapse_icon_color`) because the generator preserves the JS prop name in the Python API and Dash users expect Pythonic kwargs. `className` / `rowClassName` are the exceptions — these are passed straight through to `react-arborist`. Keep new props snake_case unless they're a direct passthrough to a camelCase library prop.

### Demo vs. Dash runtime

`src/demo/App.js` simulates Dash's runtime by maintaining local state and passing a hand-rolled `setProps` to `<Tree>`. The real `setProps` is injected by Dash at runtime. So if you add a new prop that the component writes back (i.e. a controlled prop reported to Dash callbacks), make sure the demo's `setProps` stub still works.

### Bundle-asset wiring

`dash_tree_components/__init__.py` builds the `_js_dist` list that tells Dash which JS files to serve. If you add a new chunk (e.g. via webpack `splitChunks` async groups or a CSS file), it must be:

1. Added to `_js_dist` (or `_css_dist`) in `__init__.py`,
2. Listed in `MANIFEST.in` so it's included in the sdist/wheel.

`_validate_init.py` enforces both at publish time.

### Tree data shape

`data` is a list of nodes. Each node has `id` (string, required by react-arborist), `name` (display text), optional `children` (array — presence determines leaf vs. folder), and optional `href` (rendered as a link for leaf nodes). Leaf nodes also support an `icon_color` override (see `Node.react.js`). See `usage.py` and `src/demo/data.js` for examples.
